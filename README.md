# cycle-phase-benchmark

An open, reproducible benchmark and baseline model for predicting menstrual cycle phase from
wearable and self-reported daily signals, evaluated against hormone-validated labels. Built for
Hack-Nation's *Foundation Models for Women's Hormonal Health* challenge (6th Global AI Hackathon).

The deliverable is a reusable scientific asset, not just a prototype: an open dataset pipeline,
a frozen participant-grouped benchmark split, a documented baseline model + evaluation harness,
explainability, and a thin inference API.

## Problem

**Task**: wearable-based 4-class menstrual cycle-phase prediction. Given one day's Fitbit +
Dexcom CGM + self-report signals for a participant, predict which phase that day falls in:

```
0 = Menstrual   1 = Follicular   2 = Fertility   3 = Luteal
```

Labels come from `hormones_and_selfreport.phase`, itself derived from hormone measurements (LH,
estrogen, PDG) taken with a Mira fertility device — i.e. the labels are hormone-validated, not
self-reported guesses. See `reports/benchmark_card.md` for the full task definition and
`reports/feature_dictionary.md` for every feature's source, units, and any data-quality fix
applied.

## Data + access note

Built on **mcPHASES** (PhysioNet): 23 tables of Fitbit, Dexcom CGM, Mira hormone, and
self-reported symptom data for 42 participants, keyed by `id` + `day_in_study`.

**mcPHASES is PhysioNet-restricted.** It requires signing a Data Use Agreement; raw data is
**not redistributed** in this repository (`data/raw/` and `Dataset/` are both gitignored). To
reproduce:
1. Sign the PhysioNet DUA and download mcPHASES.
2. Place the 23 CSVs directly in `data/raw/` (flat, no subfolder).
3. Run the pipeline below.

## Results

**Headline: the model scores 0.37 macro-F1 on the frozen held-out test split, confirmed by a
5-fold cross-validation mean of 0.366 ± 0.029 — and it beats the personal-majority baseline (the
harder of the two) in every single fold.** The close match between the single test split and the
5-fold mean is the important part: it means 0.37 isn't a lucky draw of 7 participants, it's a
stable estimate.

### Held-out test split (7 participants, 886 participant-days)

From `reports/eval_test.json`:

| approach | macro-F1 | Menstrual | Follicular | Fertility | Luteal |
|---|---|---|---|---|---|
| **model** | **0.368** | 0.377 | 0.376 | 0.238 | 0.479 |
| personal-majority | 0.280 | 0.000 | 0.389 | 0.264 | 0.465 |
| global-majority | 0.119 | 0.000 | 0.000 | 0.000 | 0.475 |

### 5-fold participant-grouped cross-validation (all 42 participants)

From `reports/crossval.json` — mean ± std macro-F1 across folds:

| approach | mean macro-F1 | std |
|---|---|---|
| **model** | **0.366** | 0.029 |
| personal-majority | 0.233 | 0.038 |
| global-majority | 0.126 | 0.007 |

Per-fold model vs. personal-majority (model wins every fold): 0.394 vs 0.267, 0.375 vs 0.255,
0.356 vs 0.203, 0.386 vs 0.183, 0.320 vs 0.257.

**Two required naive baselines**, per the benchmark card: **global-majority** (predict the single
most frequent phase from train) and **personal-majority** (predict each participant's own most
common phase, leave-one-out within the split — the harder baseline, since it already captures
"this person tends to run luteal-heavy"). A third baseline, a calendar/day-within-cycle
heuristic, was scoped out — see `reports/benchmark_card.md` for why and a concrete recipe for
building it as future work.

Explainability (`reports/shap_global.png`, `reports/shap_by_class.png`,
`reports/shap_importance.json`) and a confusion matrix (`reports/confusion_test.png`) are
generated by `src/explain.py` and `src/evaluate.py` respectively — see Limitations below for
what the top SHAP drivers reveal about a real confound in this model.

## Reproduce

### Environment setup

```
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

**macOS without Homebrew**: LightGBM's (and SHAP's LightGBM integration's) prebuilt wheel
dynamically links against `libomp.dylib` at runtime, which isn't bundled in the wheel and isn't
present without Homebrew's OpenMP package. If `import lightgbm` fails with
`Library not loaded: @rpath/libomp.dylib`, pick one:

- **Simplest, if you have or can install Homebrew**: `brew install libomp`, then run everything
  normally (no extra env var needed).
- **Simplest, if you use conda/mamba**: `conda install -c conda-forge lightgbm` installs a build
  that resolves its own OpenMP dependency inside the conda environment.
- **No Homebrew, no conda** (how this repo's own dev environment was set up): pull `libomp.dylib`
  directly from the conda-forge package, without installing Homebrew or conda, and point
  `DYLD_LIBRARY_PATH` at it for every LightGBM-dependent command:
  ```
  mkdir -p .venv/native-libs
  curl -sL "https://api.anaconda.org/download/conda-forge/llvm-openmp/22.1.8/osx-arm64/llvm-openmp-22.1.8-hc7d1edf_0.conda" -o /tmp/llvm-openmp.conda
  cd /tmp && unzip -o llvm-openmp.conda -d unzipped && cd unzipped
  pip install zstandard  # one-time, only needed to decompress the .conda package below
  python3 -c "import zstandard; d=zstandard.ZstdDecompressor(); f=open('pkg-llvm-openmp-22.1.8-hc7d1edf_0.tar.zst','rb'); open('pkg.tar','wb').write(d.stream_reader(f).read())"
  tar -xf pkg.tar lib/libomp.dylib
  cp lib/libomp.dylib <repo-root>/.venv/native-libs/libomp.dylib
  ```
  Then prefix every command below with `DYLD_LIBRARY_PATH="$(pwd)/.venv/native-libs"`, as shown.
  **`.venv/` is gitignored**, so every fresh clone/machine needs to redo this once (or use
  `brew`/`conda` instead — those don't need the env var at all).

### Pipeline (run in this order)

```
DYLD_LIBRARY_PATH="$(pwd)/.venv/native-libs" python3 -m src.features
DYLD_LIBRARY_PATH="$(pwd)/.venv/native-libs" python3 -m src.splits
DYLD_LIBRARY_PATH="$(pwd)/.venv/native-libs" python3 -m src.train
DYLD_LIBRARY_PATH="$(pwd)/.venv/native-libs" python3 -m src.evaluate --split test
DYLD_LIBRARY_PATH="$(pwd)/.venv/native-libs" python3 -m src.crossval
DYLD_LIBRARY_PATH="$(pwd)/.venv/native-libs" python3 -m src.explain --split test
DYLD_LIBRARY_PATH="$(pwd)/.venv/native-libs" python3 scripts/make_feature_stats.py
```

| step | writes |
|---|---|
| `src.features` | `data/processed/dataset.parquet`, `reports/feature_dictionary.md` |
| `src.splits` | `data/processed/splits.json`, `reports/benchmark_card.md` |
| `src.train` | `models/lgbm.txt`, `reports/train_metrics.json` |
| `src.evaluate` | `reports/eval_test.json`, `reports/confusion_test.png` |
| `src.crossval` | `reports/crossval.json` |
| `src.explain` | `reports/shap_global.png`, `reports/shap_by_class.png`, `reports/shap_importance.json` |
| `scripts/make_feature_stats.py` | `models/feature_stats.json` (population mean/std for on-the-fly `_pz` fill in `/predict`) |

Run the API locally (see `api/main.py`):

```
DYLD_LIBRARY_PATH="$(pwd)/.venv/native-libs" python3 -m uvicorn api.main:app --reload
```

Run the full test suite (tiny synthetic fixtures, no real data or model required):

```
DYLD_LIBRARY_PATH="$(pwd)/.venv/native-libs" python3 -m pytest -q
```

## Limitations

- **Small, single-cohort dataset**: 42 participants, a Canadian cohort. Findings may not
  generalize to other populations, age ranges, or ethnicities.
- **Wide train/val gap**: train macro-F1 0.814 vs. val macro-F1 0.383 (`reports/train_metrics.json`),
  even after regularizing for ~28 training participants (shallow trees, L1/L2, feature/row
  subsampling, slow learning rate). With this few participants, some overfitting to individual
  physiological quirks is expected; the 5-fold CV std (±0.029) is the more trustworthy
  generalization signal than any single fold's train/val gap.
- **A documented confound**: `demographic_vo2_max` and its per-participant-normalized twin are
  among the top-6 global SHAP drivers (`reports/shap_importance.json`). VO2max is a relatively
  stable personal trait, not a daily physiological signal — its importance suggests the model is
  partly learning "which participant is this" (a form of participant-identity leakage through a
  static trait) rather than purely the daily hormonal signature. This is disclosed, not hidden;
  a stricter benchmark variant might exclude static per-participant features entirely.
- **Fertility is the hardest class to separate** (F1 0.238 on test, the lowest of all four),
  consistent with it being the shortest phase and adjacent to Follicular in the cycle — the two
  are the most commonly confused pair per `reports/confusion_test.png`.
- **Glucose mixed-unit fix applied**: raw `glucose_value` mixes mmol/L and mg/dL readings (a
  known data-quality issue found during EDA, not disclosed in the mcPHASES README); values < 30
  are treated as mmol/L and converted. See `reports/feature_dictionary.md` for the exact rule.
- **No imputation**: missing values are left as NaN throughout (LightGBM handles them natively).
  Coverage varies by signal — see `reports/eda.md` for per-signal missingness.

## Future work

- **Pool more participants across studies** — the shared-infrastructure argument this challenge
  is built around: a bigger, multi-cohort version of this benchmark would directly address the
  small-n overfitting above, and is exactly the kind of reusable asset meant to outlast this
  hackathon.
- **Temporal / sequence models** (e.g. a GRU/transformer over each participant's day sequence,
  or the day-within-cycle calendar heuristic scoped out of `reports/benchmark_card.md`) instead
  of treating each day independently.
- **Extend the same evaluation protocol to NHANES menopause-stage prediction**, per the original
  build plan's fallback problem — same architecture (grouped splits, baselines, SHAP, API), a
  different hormonal life stage, testing whether this infrastructure generalizes across the
  hormonal lifespan, not just within one dataset.
